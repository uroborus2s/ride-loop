# 需求摘要

- `REQ-001` P0 统一账户与角色模型：同一司机账户同时支持接单、分销、司机专区消费。
- `REQ-002` P0 二维码归因：司机二维码建立乘客归因窗口，商城和出行支付都可命中分销收益。
- `REQ-003` P0 商城下单与支付：乘客可在武汉单城完成商品浏览、下单、支付。
- `REQ-004` P0 返佣与站内钱包：支付成功后生成佣金台账、站内余额或返现券；退款走逆向分录。
- `REQ-005` P0 司机专区消费：司机使用站内余额和返现券购买早餐、充电、洗车等服务。
- `REQ-006` P0 叫车与行程订单：乘客在武汉单城创建出行订单。
- `REQ-007` P0 派单调度：Go 调度服务管理在线态、匹配、重派和接单反馈。
- `REQ-008` P0 完单支付与收益拆分：履约司机和引流司机可分离记账。
- `REQ-009` P0 运营后台：管理城市、司机、商品、规则、订单和台账。
- `REQ-010` P1 审计与对账：支付、返佣、退款、派单关键节点可追踪。
- `REQ-011` P0 司机入驻与审核：司机资料、资质、能力启停与审核记录。
- `REQ-012` P1 商家/供应商履约：条件性支持第三方履约、核销和售后协同。
- `REQ-013` P0 消息与触达：订单、收益、派单通知；司机接单不只依赖微信订阅消息。
- `REQ-014` P1 客服、售后与申诉：工单、退款、投诉和收益申诉。
- `REQ-015` P1 调度运营与人工干预：监控派单异常、人工处理订单。
- `REQ-016` P1 风控、审计与合规：关键动作留痕与异常行为监测。
- `REQ-017` P1 数据分析与运营看板：拉新、出行、返佣、复购指标看板。

- `NFR-001` 性能：核心 API `P95 < 500ms`，派单候选返回 `P95 < 300ms`，支付回调到入账 `P95 < 3s`。
- `NFR-002` 安全：统一鉴权、支付回调幂等、后台权限控制。
- `NFR-003` 可用性：调度失败不污染主订单事实，关键错误可追踪。
- `NFR-004` 可扩展性：模型保留城市维度，但首版只落武汉。

## 端形态补充

- 乘客端：微信小程序
- 司机轻端：微信小程序，负责分销、收益、司机专区和资料
- 司机重端：独立 App，负责在线接单、定位和行程推进
- 运营端：统一管理 Web，承接运营、客服、财务、风控、调度运营

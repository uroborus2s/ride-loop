## 1. WBS 总览

- Epic：武汉单城双主线 MVP
- 目标：在不跳过软件工厂工作流的前提下，建立可进入实施阶段的任务结构
- 范围：统一账户、商城闭环、出行闭环、后台审计、联调回归

## 2. WBS 分层

### `WBS-001` 工程与治理基线

- 对应任务：`TASK-001`
- 交付目标：固化 monorepo 入口、服务骨架、运行脚本和开发治理约束
- 前置依赖：需求与设计文档确认
- 验收方式：工程目录、规范入口、基础设施约束和本地联调入口齐备

### `WBS-002` 统一账户与司机能力域

- 对应任务：`TASK-002`
- 交付目标：建立统一用户、司机档案、角色和能力状态边界
- 前置依赖：`WBS-001`
- 验收方式：账户与司机能力模型、核心接口和状态流转与设计一致

### `WBS-003` 商城与资金内循环闭环

- 对应任务：`TASK-003`, `TASK-004`
- 交付目标：完成二维码归因、商城下单支付、返佣入账和司机专区复购
- 前置依赖：`WBS-001`, `WBS-002`
- 验收方式：司机分销 -> 乘客支付 -> 返佣/券 -> 司机专区消费闭环可验证

### `WBS-004` 出行与调度闭环

- 对应任务：`TASK-005`, `TASK-006`
- 交付目标：完成叫车订单、调度匹配、接单履约和完单收益拆分
- 前置依赖：`WBS-001`, `WBS-002`
- 验收方式：乘客叫车 -> 派单 -> 司机接单 -> 行程完结 -> 支付结算链路可验证

### `WBS-005` 后台运营、审计与回归

- 对应任务：`TASK-007`, `TASK-008`
- 交付目标：完成后台配置、异常追踪、联调回归和缺陷收敛
- 前置依赖：`WBS-003`, `WBS-004`
- 验收方式：后台可观察、关键台账可追踪、双主线回归用例可执行

## 3. 关键依赖关系

1. `WBS-001` 是全部实施工作的统一入口。
2. `WBS-002` 完成前，不进入商城资金和出行调度的核心实现。
3. `WBS-003` 与 `WBS-004` 可以并行推进，但共用统一账户和订单真相边界。
4. `WBS-005` 以双主线闭环可运行作为前提，不提前替代业务主链路实现。

## 4. 准入约束

- 需求、设计、测试计划、部署说明已存在。
- 技术选型和模块边界文档已被确认。
- 当前 WBS 仅用于进入实施前的治理与排期，不视为代码承诺。

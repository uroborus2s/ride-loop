# 实施计划

**项目名称：** 千乘坊（ride-loop）  
**文档状态：** 草稿  
**负责人：** AI 软件工厂  
**主要读者：** 项目负责人 | 架构 | 开发 | QA  
**上游输入：** PRD | 系统架构 | 模块边界 | 测试计划  
**下游输出：** 任务拆分 | 迭代计划 | 编码实施  
**关联 ID：** `TASK-001` - `TASK-008`  
**最后更新：** 2026-03-29  

## 1. 实施目标

- 在不跳过工作流的前提下，为编码阶段准备清晰的服务骨架、接口契约和测试目标。
- 先打通武汉单城双主线的最小闭环，再补后台查询与审计。

## 2. 分阶段计划

| 阶段 | 目标 | 交付物 |
|---|---|---|
| Phase 0 | 固化文档与工程入口 | 文档体系、`.factory`、仓库基础结构 |
| Phase 1 | 主业务基础能力 | 统一账户、商品/司机/城市基础模型 |
| Phase 2 | 商城闭环 | 归因、商城下单支付、返佣、司机专区消费 |
| Phase 3 | 出行闭环 | 叫车、调度、接单、完单支付、收益拆分 |
| Phase 4 | 运营与审计 | 后台配置、台账查询、异常追踪 |

## 3. 候选任务

| 任务 ID | 任务名称 | 目标 | 估算（人天） |
|---|---|---|---|
| `TASK-001` | 工程初始化与依赖落地 | 固化 monorepo、Stratix 服务骨架、Go 调度骨架 | 1.5 |
| `TASK-002` | 统一账户与司机档案 | 完成身份域与权限边界 | 1.5 |
| `TASK-003` | 商城与归因闭环 | 完成二维码归因、商品与商城订单 | 2.0 |
| `TASK-004` | 返佣与司机专区 | 完成佣金台账、钱包、券和专区订单 | 2.0 |
| `TASK-005` | 叫车与行程订单 | 完成叫车入口、行程状态机 | 2.0 |
| `TASK-006` | 调度服务 MVP | 完成在线态、匹配、重派和回传 | 2.0 |
| `TASK-007` | 后台配置与审计 | 完成规则管理、订单/台账查询 | 1.5 |
| `TASK-008` | 联调与回归 | 完成双主线联调、测试与缺陷修复 | 2.0 |

## 4. 先后顺序

1. 先完成 `TASK-001` 和 `TASK-002`，保证统一账户和工程入口稳定。
2. 商城线优先做 `TASK-003` 与 `TASK-004`，因为它更快形成资金内循环闭环。
3. 并行推进 `TASK-005` 与 `TASK-006`，以打通出行链路。
4. 最后补 `TASK-007` 和 `TASK-008`，确保运营可观察、系统可回归。

## 5. 进入编码阶段的准入条件

- 需求文档、设计文档、测试计划、部署说明已存在且可用。
- 技术选型中关于 Stratix 实际安装版本的差异已补充说明。
- 核心接口和数据模型无阻塞性未决项。

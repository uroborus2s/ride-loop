## 1. 文档用途

- 明确本次首版开发工作量的范围、边界、阶段付款建议和验收建议。
- 将第三方服务、资质申请和实际资源消耗与开发服务费拆开，避免口径混淆。

## 2. 评估摘要

### 2.1 项目范围摘要

- 本次评估基于武汉单城首版上线目标。
- 首版同时覆盖“本地生活商城闭环”和“网约车派单闭环”两条主线。
- 统一司机账户，司机同时具备接单司机、分销商和商城消费者能力。
- 返现与返现券仅支持平台内循环，不支持提现。
- 技术范围为 `Node.js + Stratix` 主业务后端、`Go` 调度服务、乘客微信小程序、司机轻端微信小程序、司机重端 Android App、运营 Web 后台。

## 6. 验收节点建议

### 6.1 阶段验收节点

| 验收节点          | 对应里程碑 | 核心验收内容                                       | 主要证据                         |
| ----------------- | ---------- | -------------------------------------------------- | -------------------------------- |
| `A1` 基线验收     | `M1`       | 工程基线、账户域、司机准入流程可演示               | 环境说明、接口联调记录、页面演示 |
| `A2` 商城闭环验收 | `M2`       | 扫码归因、商城下单、支付后返佣、司机专区消费可演示 | 测试单、台账记录、页面录屏       |
| `A3` 出行闭环验收 | `M3`       | 叫车、派单、接单、行程推进、完单支付可演示         | 调度日志、订单流转记录、录屏     |

| `A5` 上线验收     | `M5`       | 测试通过、上线清单齐备、回滚预案齐备               | 测试报告、上线检查表、验收单     |

### 6.2 验收原则

- 以当前已确认的需求和设计文档作为功能验收基线。
- 验收以“主流程可用、关键页面可走通、关键接口可联调、关键异常可处理”为主，不以视觉高定制为默认标准。
- 第三方外部服务若未按时准备，导致功能无法在真实环境验证，不计为研发未完成，应在验收备注中单列。

## 7. 变更控制规则

### 7.1 视为变更的典型情形

- 从武汉单城扩展到多城市
- 新增 iOS 司机端或乘客端原生 App
- 新增提现、银行卡结算、分账、对公结算
- 新增商家独立端、商家结算中心
- 新增高级 BI、复杂营销裂变、复杂风控规则引擎
- 对已确认页面、状态机、接口契约做大范围重构

### 7.2 变更处理建议

- 变更需求进入评审后，按新增人天、影响模块、对当前排期的影响三项重新评估。
- 建议在 `2 个工作日` 内给出影响说明和补充说明。
- 未经书面确认的新增需求，不纳入当前合同交付范围。

## 9. 风险与前提

- 当前工作量评估以前期需求和设计已冻结为前提；若范围继续扩大，报价和工期需同步调整。
- 若客户不能按里程碑提供账号、资质、域名、支付、地图等外部条件，会直接影响联调和验收时间。
- 若首版要求提高到“高并发正式商用 + 多城市复制 + 完整风控合规”，应视为下一阶段项目，不建议压缩进当前首版。

## 10. 新增需求影响说明

- 新增需求：司机返现券支持通过微信或手机号转赠给其他司机。
- 影响范围：资金与券服务、司机轻端页面与消息触达、审计与回归测试。
- 建议口径：把它视为当前基线后的增量需求，单独补充报价与排期，而不是直接吞入原 `164 人天` 基线。
- 建议增量估算：`+3.0 ~ +4.0 人天`
  - 资金与券模型、接口、审计：`+2.0`
  - 司机轻端页面与消息联动：`+0.5 ~ +1.0`
  - 回归测试与验收补充：`+0.5 ~ +1.0`
- 建议工期影响：如资源不变，预计新增 `2 ~ 3` 个工作日缓冲。

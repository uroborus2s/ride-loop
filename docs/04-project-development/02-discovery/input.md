## 1. 原始任务输入

- 项目名称：`ride-loop`，中文名“千乘坊”。
- 核心创意：出租车司机既是网约车司机，也是分销商和商城消费者；通过商品专属二维码把乘客导入小程序消费，乘客获得打车和商城优惠，司机获得支付后返佣，并在司机专区完成平台内复购。
- 出行业务要求：系统包含线上网约车派单调度能力。
- 技术栈倾向：前端为微信小程序，后端为 `Node.js + Stratix`，调度服务希望使用 `Go` 或 `Rust`，数据库为 `PostgreSQL + Redis`，后台管理采用 Stratix Web 管理脚手架。

## 2. 已确认输入

- 初始化边界：采用“多服务”方案，而不是单体。
- 调度服务语言：`Go`。
- 首版范围：商城与出行双主线同时进入 MVP，不做单线试点。
- 身份模型：统一司机账户；出租车司机既是接单司机，也是分销商，同时也是司机专区消费者。
- 资金模型：返现和返现券只能站内循环，不支持提现。
- 运营范围：武汉单城市先跑通。

## 3. 当前约束

- 当前环境尚未安装本地 Go 工具链。
- 当前环境未确认可用的 Stratix 公开版本与技能版本锚点一致。
- 按软件工厂工作流，需求与设计文档完成并确认前，不进入代码实现阶段。

## 4. 当前结论

- 架构方案暂定为“方案 A：双核服务 + 共享平台库”。
- 需要在需求和设计阶段明确统一账户、归因窗口、站内钱包、退款冲抵、调度短态与订单真相分离等关键规则。

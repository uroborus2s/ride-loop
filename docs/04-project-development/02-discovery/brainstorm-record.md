## 1. 背景与问题

- 当前背景：出租车司机既掌握线下乘客触达入口，又是本地城市高频刚需服务的消费者，但传统分销、商城和出行系统彼此割裂。
- 触发原因：用户希望构建一个兼顾商城与网约车派单的 O2O 闭环平台，通过司机二维码导流、支付后返利和司机专区复购，把资金沉淀在平台内部。
- 需要解决的核心问题：如何在同一平台中同时跑通“商城购买闭环”和“出行叫车闭环”，并让司机既是履约者又是分销者，同时保持返利不可提现。

## 2. 用户意图

- 想达到什么结果：初始化一个遵循“山海工枢”阶段工作流的项目，先完成文档，再进入实现。
- 为什么现在做：需要先把业务范围、技术方向、服务边界和资金规则固定下来，避免一开始就把项目做成过重的微服务系统。
- 成功的直观判断：第一版文档能够支撑后续按阶段推进，并明确商城、出行、返佣、钱包、派单和后台的边界。

## 3. 已知约束

- 时间约束：优先形成可进入实现阶段的文档闭环，而不是先写代码。
- 技术约束：后端主业务使用 `Node.js + Stratix`，派单服务使用 `Go`，主数据库为 `PostgreSQL`，缓存与高并发短态使用 `Redis`。
- 资源约束：当前环境没有本地 `factory-*` 命令、没有本地 `go` 命令，且公开可安装的 Stratix 版本与技能描述存在差异。
- 不做的范围：多城市、提现、复杂营销玩法、全国统一调度池。

## 4. 方案探索

### 方案 A：双核服务 + 共享平台库

- 核心思路：`gateway-api` 统一接入，`platform-core` 统一承载主业务事实，`dispatch-engine` 独立负责调度。
- 优点：多服务但不过度拆分，最适合双主线 MVP。
- 缺点：后续若商城和出行同时爆发，`platform-core` 会成为拆分热点。
- 适用条件：需要快速形成第一版业务闭环，并保持账务事实集中。

### 方案 B：三域服务 + 独立资金域

- 核心思路：商城/出行与钱包分域，形成 `commerce-core`、`account-wallet`、`dispatch-engine`。
- 优点：账务边界清晰，利于后续合规和审计。
- 缺点：首版跨服务协调成本高，支付、退款和返佣路径更复杂。
- 适用条件：资金域已经有明确的独立团队或合规约束。

### 方案 C：全面领域微服务

- 核心思路：按身份、商城、出行、钱包、调度分别拆分服务。
- 优点：长期可扩展性最好。
- 缺点：对首版过重，文档、联调、运维和基础设施成本过高。
- 适用条件：已有成熟团队和中台能力。

### 推荐方案

- 结论：采用方案 A。
- 推荐理由：在满足“多服务 + 调度独立”要求的同时，保持首版范围可控，便于先做出武汉单城双主线的可运行 MVP。

## 5. 已确认关键决策

- 采用多服务架构，不使用单体首发。
- 调度服务语言选择 `Go`。
- 第一版同时支持商城主线和出行主线。
- 司机身份统一，一个账户同时具备接单、分销和商城消费三种能力。
- 返现和返现券只能平台内循环，不支持提现。
- 先支持武汉单城市。

## 6. 未决问题

- 具体支付通道、地图与定位供应商尚未在本轮确定。
- 管理后台是否直接采用公开可用的 Stratix Web 脚手架，需在进入实现前根据可安装版本确认。
- 归因窗口的精确时长、佣金比例与城市运营规则需要后续运营配置文档补充。

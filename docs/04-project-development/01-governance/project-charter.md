## 1. 项目背景

- 当前业务背景：出租车司机天然具备高频线下触达乘客的场景，但传统出租车司机只能获得单次运费收入，缺少可持续的平台内增量收益。
- 当前痛点：本地生活商城和出行业务通常各自独立，司机导流、乘客复购、平台返利、刚需服务消费没有形成闭环，支付后收益也难以留存在平台生态内。
- 启动原因：构建“出租车司机分销 + 本地生活商城 + 网约车派单”的 O2O 闭环平台，在武汉单城先跑通司机引流、乘客消费、支付返利和司机复购的双主线 MVP。

## 2. 目标与成功标准

- 目标 1：跑通商城闭环，验证“司机二维码分销 -> 乘客消费 -> 支付后返佣 -> 司机站内复购”路径。
- 目标 2：跑通出行闭环，验证“乘客叫车 -> 派单调度 -> 行程支付 -> 优惠/分销联动”路径。
- 目标 3：统一司机身份，使同一司机既能接单，又能分销，还能消费司机专区。

| 指标 | 目标值 | 度量方式 |
|---|---|---|
| 双主线 MVP 可联调 | 达成 | 武汉单城环境下完成至少 1 条商城支付闭环和 1 条出行支付闭环 |
| 支付后返佣入账正确率 | 100% | 对账样例与台账校验 |
| 司机账户统一建模 | 达成 | 同一司机账户可完成接单、分销、站内消费三类操作 |
| 平台内循环约束 | 达成 | 返现余额与返现券均不可提现，只能站内消费 |

## 3. 范围与非目标

### 范围内

- 武汉单城市运营模型
- 统一司机账户域
- 司机二维码归因和乘客拉新
- 商城商品浏览、下单、支付、返佣、司机专区消费
- 乘客叫车、派单调度、接单、行程完成、支付、优惠联动
- 佣金台账、站内钱包、返现券、退款冲抵
- 运营后台的基础配置、订单、台账与司机管理能力

### 范围外

- 多城市运营与跨城调度
- 司机提现与对公打款
- 第三方开放平台 API 生态
- 复杂营销裂变玩法
- 动态定价、顺风车、拼车等高级调度模式

# 技术选型与工程规则

**项目名称：** 千乘坊（ride-loop）  
**负责人：** AI 软件工厂  
**主要读者：** 架构 | 开发 | 测试 | 维护者  
**上游输入：** PRD | 需求分析 | 风险登记册  
**下游输出：** 架构设计 | 开发实施 | 测试计划  
**最后更新：** 2026-03-29  

## 1. 技术画像摘要

- 技术栈：
  - 乘客端：微信小程序 + TypeScript，首版建议使用原生小程序框架并配合成熟 UI 组件库。
  - 司机轻端：微信小程序 + TypeScript，承接分销、收益、司机专区和资料。
  - 司机重端：Android 优先的独立司机 App，承接在线接单、定位和行程状态推进。
  - 接入层与主业务：`Node.js + Stratix`。
  - 派单调度：`Go`。
  - 数据层：`PostgreSQL + Redis`。
  - 后台：Web 管理端，优先使用 Stratix Web 管理脚手架；若公开可安装版本不满足要求，则用独立前端工作区承接后台视图层。
  - 商家/供应商端：条件性 Web/H5。
- 选型目标：
  - 保持商城与出行业务事实集中，先跑通双主线闭环。
  - 把高实时性的调度逻辑独立出来，降低后续扩容成本。
  - 以文档与契约驱动多服务协作，而不是先做重型微服务体系。
- 适用范围：武汉单城首版 MVP 及其之后的单城扩展阶段。

## 2. 必装/必选模块

- `gateway-api`：统一接入层，承接小程序和后台 API。
- `platform-core`：Stratix 主业务服务，包含身份、商城、出行订单、返佣与运营配置。
- `dispatch-engine`：Go 调度服务，承接司机在线态和派单计算。
- `driver-app`：司机重端，承接实时履约动作。
- `PostgreSQL`：主业务、台账、订单、配置的最终事实源。
- `Redis`：会话、幂等、券、高并发短态、司机在线态缓存。
- `Docker Compose`：本地联调基础设施编排。

## 3. 工程规则

- 文档优先：进入编码前必须阅读 `technical-selection.md`、`system-architecture.md`、`module-boundaries.md`。
- 单仓多服务：所有服务共用一个仓库，统一目录、版本管理和文档入口。
- 接入层统一：所有外部流量先进入 `gateway-api`，不直接打到内部服务。
- 订单真相集中：商城订单、出行订单、支付、台账都以 `platform-core + PostgreSQL` 为准。
- 调度状态分离：`dispatch-engine` 负责匹配和派单短态，不拥有最终订单事实。
- 数据访问分层：主业务侧遵循 `controller -> service -> repository`，服务层不得绕过 repository 直接访问数据库。
- 资金规则刚性：返现余额与返现券只可站内消费，不支持提现。
- 配置优先：归因窗口、佣金规则、城市配置、司机专区商品必须可配置，不写死到业务逻辑。
- 容器优先：由于当前环境没有本地 Go 工具链，调度服务开发与联调优先采用容器化运行。
- 双端分工：司机分销/收益与司机接单/定位分属轻端和重端，不在一个终端强行混载。

## 4. 管理后台/运营侧要求

- 首版后台必须覆盖城市配置、司机档案、商品、活动、佣金规则、订单查询、台账查询、异常核查。
- 后台权限至少区分城市运营、调度运营、客服、财务/对账、风控/审计、技术支持六类角色。
- 后台只承载管理和查询，不承载核心账务逻辑。

## 5. 版本与落地策略

- 设计目标以 Stratix 生态为准，主业务服务按 Stratix 的配置、DI 和资源分层方式设计。
- 当前公开可安装的 Stratix 版本与技能中的版本锚点存在差异。
- 进入编码前必须新增一条 ADR，明确“设计目标版本”和“实际安装版本”的映射关系，避免实现阶段偏离设计。

## 6. 同步要求

- 进入实现前必须阅读本文件。
- 技术栈变更后同步更新设计、测试和 `.factory/memory/`。

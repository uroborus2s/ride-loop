## 1. 本轮确认的设计决策

| 决策 ID | 结论 | 设计影响 |
|---|---|---|
| `UX-DEC-001` | 司机专区支持“到店核销”和“配送/到店自提”两种履约模式并存，但单个商品只能选择一种履约模式 | 商品卡、详情页、结算页必须展示履约标签与不同表单 |
| `UX-DEC-002` | 司机重端首版明确为 Android 优先，不设计 iOS 差异功能 | 交互可以基于 Android 手势、权限模型、通知与后台运行能力设计 |
| `UX-DEC-003` | 运营 Web 首版做到“菜单级 + 页面动作级”，对审核、退款、调度干预、权限变更等高风险按钮做到动作级权限 | 页面内按钮显隐、可点状态与操作原因弹窗都必须按权限渲染 |
| `UX-DEC-004` | 退款后负余额需要单独输出资金状态图，并在司机端与后台都显式展示 | 收益页、台账页、退款页、风控/客服工单都要能识别负余额与冲抵顺序 |

## 2. 整体设计原则

- 乘客端强调“优惠可见、路径最短、无需理解分销逻辑”。
- 司机轻端强调“收益透明、分销简明、司机专区复购顺手”。
- 司机重端强调“高对比、低分心、状态明确、误触成本低”。
- 运营 Web 强调“高信息密度、强筛选、强审计、强权限提示”。
- 全端统一使用明确状态词，不使用模糊运营话术。

## 4. 乘客微信小程序交互设计

### 4.1 产品定位与风格建议

- 风格方向：城市本地生活 + 出行优惠入口，不做重商城首页。
- 视觉重点：
  - 扫码落地页直接展示“司机专属优惠”和两个主入口。
  - 首页主 CTA 只保留两个：`去购物`、`去叫车`。
  - 优惠信息用橙色强调，信任信息和支付安全用深青绿色。
- 不建议：
  - 把分销逻辑、佣金术语暴露给乘客。
  - 首屏堆满活动浮层、抽奖轮盘、复杂运营贴片。

### 4.2 导航建议

- Tab 结构：
  - 首页
  - 商城
  - 出行
  - 我的
- 首页顶部结构：
  - 城市与定位
  - 扫码优惠 banner
  - 商城快捷入口
  - 打车快捷入口
  - 热门商品与司机推荐位

### 4.3 关键页面与交互

| 页面 | 主要模块 | 主交互 | 异常态 |
|---|---|---|---|
| 扫码落地页 | 优惠卡、司机背书、双入口 | 一键进商城、一键叫车、查看规则 | 归因失效时降级成普通首页 |
| 商城列表 | 分类、商品卡、专区标签 | 切类目、看详情、加入结算 | 无商品时展示附近推荐 |
| 商品详情 | 图文、价格、优惠、履约方式 | 立即购买、选择规格、查看履约说明 | 商品下架时禁止下单 |
| 叫车页 | 起终点、预估价、优惠提示 | 选地址、发起叫车、查看优惠抵扣 | 非武汉范围直接阻断 |
| 订单追踪 | 司机信息、状态时间线、客服入口 | 取消、联系客服、查看轨迹 | 派单失败时给改约/重试 |
| 我的 | 商城订单、行程订单、券、客服 | 进订单详情、发起售后 | 未登录时展示登录引导 |

### 4.4 微交互建议

- 首页和扫码落地页的优惠卡片可以做 200ms 内的轻微浮起动效。
- 叫车成功后订单状态页使用时间线刷新，不用全屏骨架屏反复闪烁。
- 支付成功页采用“双回流 CTA”：`继续叫车`、`逛司机推荐商品`。

## 5. 司机轻端微信小程序交互设计

### 5.1 产品定位与风格建议

- 风格方向：不是“司机接单台”，而是“司机经营工作台”。
- 视觉重点：
  - 数字优先：今日收益、待入账、可用余额、临期券放在首屏上半区。
  - 区块优先：分销、收益、司机专区、工单四个区块结构固定。
  - 履约标签明确：司机专区商品必须清晰标识 `到店核销`、`到店自提`、`配送`。
- 不建议：
  - 把轻端做成运营术语堆砌的表格式页面。
  - 把“接单在线状态”放进轻端首页。

### 5.2 导航建议

- Tab 结构：
  - 工作台
  - 消息
  - 我的
- 工作台内一屏四区：
  - 收益摘要
  - 分销入口
  - 司机专区
  - 待办/消息

### 5.3 关键页面与交互

| 页面 | 主要模块 | 主交互 | 设计说明 |
|---|---|---|---|
| 工作台首页 | 收益卡片、分销卡片、专区入口、待办 | 点收益、点分销、点专区、点待办 | 首屏不超过 2 次滚动完成主功能入口 |
| 入驻申请 | 基础信息、证件上传、车辆信息、银行卡 | 保存草稿、提交审核 | 按步骤拆分，顶部显示完成度 |
| 分销中心 | 扫码人数、成交额、二维码入口、素材入口 | 创建二维码、分享海报、看归因订单 | 以“今日/累计”双维度展示 |
| 收益中心 | 待入账、可用余额、临期券、负余额提示 | 看台账、看券、进专区 | 负余额用琥珀色提示而不是红色告警 |
| 司机专区列表 | 分类、履约标签、可用券提示 | 看详情、切分类、查看最近订单 | 商品卡固定展示履约模式 |
| 司机专区结算 | 履约方式、收货/到店信息、券、余额 | 选券、填地址或门店、提交 | 到店核销与配送表单动态切换 |
| 工单中心 | 工单列表、创建工单、处理时间线 | 新建、补充附件、看结论 | 默认按“处理中”优先排序 |

### 5.4 履约模式交互规则

| 履约模式 | 商品详情页展示 | 结算页展示 | 订单页展示 |
|---|---|---|---|
| 到店核销 | 门店、营业时间、核销说明 | 门店选择、联系人、手机号 | 核销码、到店时间、门店地图 |
| 到店自提 | 自提点、提货时段 | 自提点、预约时间、手机号 | 取货码、自提时间 |
| 配送 | 配送范围、配送费、时效 | 地址、联系人、手机号、配送时段 | 配送状态、骑手/门店履约信息 |

### 5.5 视觉建议

- 卡片背景建议使用浅暖白和浅绿两级背景，不要纯白大平板。
- 收益数字用深绿主色，风险提示用琥珀色，禁用态用中性灰蓝。
- 返现券组件采用“票券切边”轮廓，但保持信息清晰，不做强拟物。

## 6. 司机重端 App 交互设计

### 6.1 产品定位与风格建议

- 风格方向：Android 优先的车载履约工具，不是运营看板，也不是商城端。
- 视觉重点：
  - 夜间高可读：深石墨底色 + 高亮青绿动作色 + 橙色主 CTA。
  - 单屏单任务：派单页只服务“接/拒/看详情”，行程页只服务状态推进。
  - 安全优先：高风险操作要二次确认或长按确认。
- 不建议：
  - 在行程页混入营销信息、收益 banner、非履约通知。
  - 依赖复杂侧滑菜单或深层页面嵌套。

### 6.2 导航建议

- 主导航：
  - 工作台
  - 订单
  - 消息
  - 账户
- 高优先级页面：
  - 派单弹层为强前置
  - 当前行程页可覆盖任何普通页面

### 6.3 关键页面与交互

| 页面 | 主要模块 | 主交互 | 风险控制 |
|---|---|---|---|
| 登录与设备绑定 | 登录、换绑、版本检查 | 登录、绑定、升级 | 强制升级时不允许跳过 |
| 权限引导 | 定位、通知、后台运行 | 去授权、重试校验 | 核心权限未过不允许上线 |
| 接单工作台 | 上下线、当前状态、在途订单 | 上线、下线、进当前行程 | 在途订单期间禁止下线 |
| 派单弹层 | 倒计时、上车点、预估收入 | 接单、拒单、看详情 | 关键按钮尺寸适配驾驶场景 |
| 当前行程页 | 到达、开始、结束、联系乘客 | 状态推进、拨号、导航 | 非法状态按钮直接禁用 |
| 异常页 | 异常类型、说明、附件、介入 | 上报异常、申请取消 | 关键字段不全不可提交 |
| 历史订单 | 列表、筛选、详情入口 | 查详情、发起申诉 | 不展示复杂财务细节 |

### 6.4 交互细节建议

- 上线开关不做 iOS 弹簧式轻动效，直接使用明确的二段切换反馈。
- 派单弹层倒计时颜色逐步从青绿过渡到橙红，但数值必须常显。
- 行程推进按钮建议采用大尺寸底部吸附按钮，避免顶部小按钮误触。
- 异常页上传凭证建议支持相机直拍，减少多步选择。

### 6.5 视觉建议

- 背景：`#0B1220`
- 主操作色：`#14B8A6`
- 强提醒：`#F97316`
- 危险动作：`#E11D48`
- 字号层级：关键数字与状态标签至少比正文高 2 级

## 7. 运营 Web 交互设计

### 7.1 产品定位与风格建议

- 风格方向：武汉单城运营控制台，而不是演示型 BI 大屏。
- 视觉重点：
  - 表格、筛选、侧边详情抽屉是主形态。
  - 经营总览保留 KPI 卡，但不做炫技型大屏。
  - 高风险按钮必须显式展示权限、原因、二次确认。
- 不建议：
  - 用大量渐变背景和装饰图形稀释数据信息密度。
  - 所有模块统一成“编辑弹窗”，复杂配置应有独立详情页。

### 7.2 导航建议

- 左侧一级菜单：
  - 总览
  - 司机管理
  - 商品与活动
  - 订单中心
  - 调度中心
  - 资金中心
  - 工单中心
  - 风控审计
  - 系统设置
- 右上全局区：
  - 城市切换
  - 全局搜索
  - 待办告警
  - 当前账号与角色

### 7.3 关键页面与交互

| 页面 | 主要模块 | 主交互 | 权限控制建议 |
|---|---|---|---|
| 首页总览 | KPI 卡、异常告警、待办 | 点进异常列表、切时间范围 | 全角色可看，但字段按角色裁剪 |
| 司机审核 | 列表、详情抽屉、审核动作 | 通过、驳回、暂停能力 | 审核按钮动作级权限 |
| 商品管理 | 列表、草稿、版本、发布 | 保存草稿、发布、下线 | 发布与下线动作级权限 |
| 派单监控 | 地区筛选、订单池、司机池、异常池 | 进干预、切区域、看轨迹 | 仅调度角色可操作干预按钮 |
| 退款与售后 | 列表、详情、冲销引用 | 退款通过、退款驳回、人工调整 | 财务动作级权限 + 二次确认 |
| 工单中心 | 工单列表、处理时间线、转办 | 接单、转办、结案、重开 | 客服/财务/风控按动作分权 |
| RBAC 设置 | 角色列表、权限矩阵、用户绑定 | 新建角色、改权限、绑定账号 | 管理员动作级权限 + 审计留痕 |

### 7.4 动作级权限建议

| 动作类型 | 权限粒度 | 表现形式 |
|---|---|---|
| 查询类 | 菜单级/页面级 | 无权限不显示菜单或只显示无数据页 |
| 普通编辑类 | 页面动作级 | 无权限时按钮置灰并提示原因 |
| 高风险动作类 | 动作级 + 二次确认 | 按钮显隐 + 原因输入 + 审计提醒 |

### 7.5 视觉建议

- 背景以浅灰蓝和白色卡片为主，避免大面积纯蓝。
- KPI 卡重点数值可使用深蓝，异常数与风险数使用琥珀/玫红。
- 表格行 hover 高亮必须明显，但不能抢占状态标签颜色。

## 8. 共享交互规则

- 所有订单、工单、台账都用统一时间线组件表达状态流转。
- 状态颜色统一：
  - 进行中：青绿色系
  - 待处理：琥珀色系
  - 已完成：深绿色系
  - 已驳回/高风险：玫红或红色系
- 所有空态都要说明“为什么为空”和“下一步可以做什么”。
- 所有失败态都优先返回可执行建议，而不是只有错误码。

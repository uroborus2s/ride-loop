## 1. 服务结构

- `gateway-api`
  - 作用：统一接入层
  - 主要能力：鉴权、限流、幂等、聚合响应
- `platform-core`
  - 作用：Stratix 主业务服务
  - 主要能力：身份、商城、出行订单、钱包、券、运营配置
- `dispatch-engine`
  - 作用：Go 调度服务
  - 主要能力：在线态、匹配、重派、接单响应

## 2. `platform-core` 设计原则

- 采用 `controller -> service -> repository` 分层。
- 身份、商城、出行、资金、运营作为逻辑子域放在同一服务中。
- 主业务数据库访问统一收敛到 repository 层。
- 订单状态机、返佣和退款冲抵优先用业务服务编排，不放在控制器。
- 复杂多表一致性逻辑在 repository 或 business repository 中收敛，不拆散到多个 service。

## 3. `dispatch-engine` 设计原则

- 使用 Go 编写独立 HTTP 服务。
- 主要包建议：
  - `cmd/dispatch-engine`：启动入口
  - `internal/http`：接口层
  - `internal/engine`：派单流程
  - `internal/matcher`：司机候选筛选
  - `internal/store`：Redis 状态读写
- 调度服务只对派单过程负责，不直接写主业务订单表。

## 4. 同步交互设计

- `gateway-api -> platform-core`
  - 协议：HTTP JSON
  - 特点：对外统一入口、标准错误码
- `platform-core -> dispatch-engine`
  - 协议：HTTP JSON
  - 特点：同步发起派单，异步/同步混合接收回传
- `dispatch-engine -> platform-core`
  - 协议：HTTP JSON
  - 特点：回传派单结果、接单、超时重派结果

## 5. 关键内部能力

- 身份能力：统一账户查询、司机角色判定、司机档案状态检查。
- 商城能力：商品与归因会话查询、商城订单状态推进。
- 出行能力：行程订单状态机、调度请求发起、订单状态回填。
- 资金能力：佣金规则命中、台账入账、退款逆向分录、券核销。
- 运营能力：城市与规则配置、订单与台账查询。

## 6. 可靠性策略

- 幂等：
  - 支付回调
  - 退款回调
  - 派单结果回传
- 重试：
  - 资金分录写入失败可重试
  - 派单超时按规则重派
- 审计：
  - 关键状态变更要带 `requestId` 与业务主键

## 7. 实现前约束

- 未确认实际可安装 Stratix 版本前，不生成最终依赖锁定文件。
- 先固化服务接口与模块边界，再生成服务骨架。
- Go 本地环境缺失不阻塞设计，但编码阶段优先采用容器化工具链。

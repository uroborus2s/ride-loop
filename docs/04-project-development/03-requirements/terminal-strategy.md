## 1. 结论先行

- 第一版除微信端外，至少还需要一个 **运营管理 Web 端**。
- 如果平台要做真实商业化的网约车派单，**司机不能只依赖微信小程序作为唯一接单端**。
- 推荐采用双司机端策略：
  - `司机轻端`：微信小程序，承接分销、收益、司机专区、资料与轻交互。
  - `司机重端`：Android 优先的独立司机 App，承接在线接单、定位上报、派单提醒、行程状态推进。
- 客服、财务、风控、调度运营在第一版不必拆成单独前端，可先沉入统一的运营管理 Web，按角色控权。
- 若第一版引入第三方履约商家，则还需要一个简化版商家/供应商 Web/H5；若是平台自营，可先由后台代履约。

## 2. 第一版建议保留的端

| 端            | 是否必需     | 推荐形态     | 核心目标                                     |
| ------------- | ------------ | ------------ | -------------------------------------------- |
| 乘客端        | 必需         | 微信小程序   | 扫码、商城、叫车、支付、订单查询             |
| 司机轻端      | 必需         | 微信小程序   | 分销二维码、收益、司机专区消费、资料提交     |
| 司机重端      | 强烈建议必需 | Android App  | 在线接单、持续定位、实时派单、行程执行       |
| 运营管理端    | 必需         | Web 管理后台 | 城市、司机、商品、规则、订单、台账、人工干预 |
| 商家/供应商端 | 条件性       | Web/H5       | 履约、核销、库存、售后协同                   |

## 3. 为什么乘客端适合用微信小程序

- 你的核心拉新入口是司机二维码，天然适合“微信扫码即进服务”。
- 乘客侧核心行为是低门槛消费与叫车，不需要长时间后台驻留。
- 商城下单、支付、优惠展示、叫车发起、订单查询都与微信支付和小程序入口高度匹配。

## 4. 司机端为什么不能只靠小程序

### 小程序能承担的司机功能

- 司机入驻资料提交和审核进度查询。
- 分销二维码生成、海报分享、归因订单查看。
- 佣金台账、站内余额、返现券和司机专区购物。
- 历史订单、收益看板、客服和反馈入口。
- 在司机主动打开小程序时，承接轻量订单辅助和订单查询。

### 小程序不适合作为唯一接单端的功能

- 长时间稳定在线接单。
- 高频实时定位上报与后台保活。
- 低延迟接收派单并强提醒。
- 行程状态推进与导航协同。
- 网络切换、系统挂起或进程回收后的履约连续性保障。

### 当前能力判断依据

- 微信提供了后台定位接口 `wx.startLocationUpdateBackground`，说明小程序在获得授权和声明后可以处理前后台位置更新，但这不等于它适合作为商业化调度终端的唯一载体。[官方 API](https://developers.weixin.qq.com/miniprogram/dev/api/location/wx.startLocationUpdateBackground.html) / [可访问镜像说明](https://www.cnblogs.com/AtlasLapetos/p/18633898)
- 微信小程序订阅消息要求用户主动订阅，长期订阅消息目前只向政务民生、医疗、交通、金融、教育等特定线下公共服务开放，更适合作为服务通知补充，而不是实时派单主通道。[官方文档](https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/subscribe-message.html) / [可访问镜像说明](https://www.cnblogs.com/cnsyear/p/13995565.html)
- 按微信小程序运行机制，普通小程序进入后台后会被挂起，长时间不回前台还可能被销毁，这对持续在线接单天然不稳定。[官方文档](https://developers.weixin.qq.com/miniprogram/dev/framework/runtime/operating-mechanism.html) / [可访问镜像说明](https://blog.csdn.net/qq_60961397/article/details/130330029)

## 5. 司机端形态判断

### 方案 A：司机只用微信小程序

- 优点：
  - 上手快
  - 安装成本低
  - 分销和收益闭环天然顺滑
- 缺点：
  - 实时派单不稳定
  - 后台定位与强提醒可靠性不足
  - 不适合正式商业运营强度
- 适用范围：
  - 小规模试点
  - 以分销商城为主，叫车为实验性能力

### 方案 B：司机小程序 + 独立司机 App

- 优点：
  - 小程序继续承接分销与收益
  - 独立 App 专门处理接单和定位
  - 更接近真实出行运力运营模式
- 缺点：
  - 首版投入更高
  - 司机需要额外安装 App
- 适用范围：
  - 要正式运营网约车派单
  - 要求持续在线、稳定定位和强提醒

### 推荐方案

- 推荐采用 **方案 B**。
- 乘客全部留在微信小程序。
- 司机分销、收益、司机专区留在微信小程序。
- 司机接单、定位、行程状态推进落在独立司机 App。

## 6. 各端功能分配

### 乘客微信小程序

- 注册登录
- 扫码落地页
- 商品浏览与下单支付
- 优惠券与活动
- 叫车、订单追踪、取消
- 行程支付
- 商城订单/出行订单查询
- 客服与售后入口

### 司机微信小程序

- 司机入驻与资料提交
- 分销二维码生成与分享
- 拉新归因订单查看
- 佣金、余额、返现券查询
- 司机专区下单与消费
- 司机账单与历史订单
- 基础消息中心
- 客服、申诉、反馈

### 司机独立 App

- 登录与设备绑定
- 在线/离线切换
- 实时定位上报
- 实时派单接收
- 抢单/接单/拒单
- 到达上车点、开始行程、结束行程
- 强提醒、语音播报
- 导航联动
- 异常上报和订单申诉

### 运营管理 Web

- 城市配置
- 司机审核与司机管理
- 商品、专区商品、活动和规则配置
- 商城订单、出行订单、司机专区订单查询
- 台账、余额、券、退款冲抵查询
- 派单监控与人工干预
- 审计日志、异常工单、客服处理

### 商家/供应商端

- 待履约订单
- 核销
- 履约状态更新
- 售后协同

## 7. 第一版端建设顺序

1. 乘客微信小程序
2. 运营管理 Web
3. 司机微信小程序
4. 司机独立 App
5. 商家/供应商端（仅在第一版引入外部履约商家时建设）

## 8. 对当前需求文档的影响

- 司机端需求必须拆成轻端与重端两组能力。
- 需要补充司机入驻、消息触达、调度人工干预、商家履约、客服售后和风控审计相关需求。
- 不能再把“司机接单”和“司机分销/收益”视为同一终端产品。
